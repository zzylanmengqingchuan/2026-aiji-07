# 《箱域擂台》(Box Rivals) 产品需求文档

**文档版本**: v0.1
**作者**: zzy
**状态**: 草案

---

## 1. 背景与目标

### 1.1 项目背景
继《坦克大战》《坦克动荡》之后的第三款小游戏。核心诉求是"对抗性 + 低建模成本"——参考《跳一跳》的轻量单人机制,但引入实时对抗;参考《推箱子》(Sokoban)的规则骨架,但把"箱子"改造成双方共享的争夺资源,从而天然产生阻挡、抢夺、卡位等对抗策略。

### 1.2 世界观设定
场地是一片被切割成方格的"仲裁场",场上散落着若干块"共鸣箱",箱子会对特定频率的能量场(玩家的"归属圈")产生响应。两名对抗者被投放进同一片场地,谁先把任意一块箱子引导进自己的归属圈,谁就完成"共鸣",赢得这一局。

### 1.3 目标
- 验证"共享资源型推箱对抗"这一核心机制的可玩性
- 用一套棋盘引擎,支撑三种对战形态,不重复造轮子
- 探索"LLM Agent 互相对战、人类观战"这一新的内容形态,作为公众号选题素材

### 1.4 非目标(本期不做)
- 不做真正的多人联机(跨设备、跨网络对战)
- 不做美术/动画精修,视觉停留在色块+网格阶段
- 不做排位、匹配、账号系统

---

## 2. 用户与场景

| 场景 | 用户 | 核心诉求 |
|---|---|---|
| 人 vs 人 | 同一设备的两名玩家 | 面对面快速对抗,规则一目了然 |
| 人 vs 机 | 单个玩家,无人陪玩 | 不需要建房间/邀请别人,随时能开一局 |
| Agent vs Agent | 观战的人类(包括开发者自己和读者) | 看两个 AI "博弈",作为内容/演示,不参与操作 |

---

## 3. 核心玩法(三种模式共用)

### 3.1 规则总览
- 棋盘为 R×C 网格,元素:墙、空地、共享箱子、P1 归属圈、P2 归属圈、两名玩家
- 每回合(或每次按键),玩家可选择上/下/左/右移动一格
- 若目标格有箱子,箱子被同向推动一格,前提是箱子前方不是墙、不是另一个箱子、不是对方玩家
- 玩家不能移动进对方玩家所在格
- 判胜:任意箱子进入某玩家的归属圈,该玩家立即获胜

### 3.2 对抗策略(设计意图,非强制规则)
- **抢箱**:抢先把离自己归属圈近的箱子推过去
- **堵路**:站在对方与目标箱子之间的必经格子上
- **弃子**:故意把箱子推进三面靠墙的死角,使其对双方都失效,拖延对方进程

### 3.3 死锁判定(P1 待定功能)
需要检测箱子被推入不可逆死角的情况(如贴墙角、或被墙+另一箱子夹死),并在界面上给出提示,避免玩家困惑"为什么这个箱子没用了"。

## 4. 分模式需求

### 4.1 模式一:人 vs 人(已完成原型)
- P1 使用 WASD,P2 使用方向键,同屏实时操作
- 无需额外开发,复用现有棋盘引擎

### 4.2 模式二:人 vs 机
**目标**:不需要建房间、不需要联网,单机启动即可开局。

**AI 设计(规则式,非 LLM)**:
1. 用 BFS 计算到"每个箱子可推动点"的最短路径
2. 对每个候选箱子打分:
   `分数 = -(到达推动点的步数) - (箱子到我方归属圈的距离) + (若箱子已接近对方归属圈,追加抢箱权重)`
3. 选分数最高的箱子作为当前目标,执行下一步
4. 兜底策略:若所有箱子均已死锁,AI 切换为"堵路模式"——移动到对手与其最近可用箱子之间

**验收标准**:
- AI 能在无外部箱子干扰情况下,独立完成推箱进圈
- AI 具备基本的抢箱/堵路行为,不是无脑单线程推进

### 4.3 模式三:Agent vs Agent(人类观战)
**目标**:两个 Claude 实例分别扮演 P1、P2,交替调用 API 决策,人类只观看,不参与操作。

**交互流程**:
1. 每回合将棋盘状态序列化为文本 Prompt,发送给当前行动方的 Agent
2. Agent 返回一个 JSON 格式的方向指令(如 `{"move": "up"}`)
3. 系统解析并执行该指令,更新棋盘,渲染下一帧
4. 循环直至分出胜负或达到最大回合数(防止死循环)

**Prompt 设计要点**:
- System Prompt:说明角色(P1/P2)、规则、输出格式限制(仅输出 JSON,不要多余文字)
- User Prompt(每回合):当前棋盘、己方/对方坐标、箱子坐标列表、双方归属圈坐标

**观战体验要求**:
- 提供"下一步"手动步进 和 "自动播放"两种模式,自动播放需可调速
- 每回合执行完展示棋盘变化,而非直接跳到终局
- 达到最大回合数仍未分胜负时,判定为平局并展示原因

**验收标准**:
- 两个 Agent 能在不越权(非法移动、幻觉坐标)的前提下完成完整对局
- 出现非法/无法解析的 Agent 输出时,有兜底处理(如判负或重试一次)

## 5. 技术方案

### 5.1 总体架构
三种模式共用同一套棋盘引擎,差异仅在"决策来源"层:

```
GameEngine(棋盘状态 + 移动/推箱/判胜逻辑)
        ├─ KeyboardAgent(人 vs 人 / 人 vs 机 的人类一侧)
        ├─ HeuristicAgent(人 vs 机 的 AI 一侧,BFS + 打分)
        └─ LLMAgent(Agent vs Agent 双方,调用 Claude API)
```

### 5.2 分阶段实现顺序
1. **阶段一**:抽出统一的 Agent 接口,把现有键盘输入包装成 `KeyboardAgent`,验证棋盘引擎可以脱离"实时按键"独立运行(即支持"引擎主动 tick、外部提供下一步"的调用方式)——这是后两个模式的前提
2. **阶段二**:实现 `HeuristicAgent`,跑通人机模式
3. **阶段三**:实现 `LLMAgent`,跑通 Prompt 序列化、API 调用、JSON 解析、非法输出兜底
4. **阶段四**:补齐模式选择界面 + Agent vs Agent 的观战播放控制条

### 5.3 关键技术点
- 棋盘引擎需要"回合制可暂停"的执行模型,而不是原来纯键盘监听的即时响应模型
- LLM 调用需要处理:格式不合法、坐标幻觉(指向不存在的位置)、超时,均应有兜底(如判负或跳过本回合)
- 需限制最大回合数,避免 Agent 对局无限进行

---

## 6. 成功指标

- 人机模式:AI 能在多局测试中稳定获胜或至少形成有效对抗(非陪跑)
- Agent vs Agent 模式:能稳定跑完一整局并产出可分享的对局记录/回放
- 内容产出:至少产出一篇围绕"用 Prompt 让两个 Claude 打推箱子擂台"的公众号素材

---

## 7. 风险与开放问题

| 风险/问题 | 说明 |
|---|---|
| LLM 决策成本 | Agent vs Agent 每回合都要调用 API,长对局可能成本偏高,需评估最大回合数上限 |
| LLM 空间推理能力 | 纯文本描述棋盘,LLM 对坐标和方向的推理是否稳定,需要小范围测试验证 Prompt 设计 |
| 死锁判定复杂度 | 判断箱子是否真正"死锁"(尤其是被多个箱子相互卡住的情况)有一定算法复杂度,可能需要分期实现 |
| 地图设计 | 当前地图为手工设计,后续若要支持多地图/编辑器,需评估工作量 |
