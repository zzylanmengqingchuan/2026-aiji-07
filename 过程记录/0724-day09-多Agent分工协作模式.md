# 多 Agent 分工协作：课程开发 Agent 统一调度模式

> 作者：cola ｜ 安吉集训 Day09 过程记录
>
> 记录一种在 Kimi Work 中通过 Agent ID 实现"主 Agent 调度多 Agent"的分工协作模式。

## 背景

集训第九天（7月24日），课程老师在腾讯会议上演示了一种多 Agent 协作模式：

**不再让每个 Agent 各自为战，而是选定一个"主 Agent"作为调度中枢，让它持有其他所有 Agent 的 ID，通过 ID 直接调用对应的 Agent 执行子任务。**

这个模式第一次见到，之前所有的 Agent 使用都是"单兵作战"——一个 Agent 干所有事，或者在多个 Agent 之间手动传话。老师演示之后，意识到 Agent 之间的协作可以做得更结构化。

## 模式说明

### 角色划分

在一个项目里划分多个专职 Agent，每个负责一个职能领域：

| Agent | 职责 |
| --- | --- |
| 课程开发 Agent | **主调度**：理解任务、拆分需求、通过 ID 调用其他 Agent |
| 版本管理 Agent | Git 提交、版本回退、变更记录 |
| 运维 Agent | 部署、服务器管理、公网访问配置 |
| GitHub Agent | 仓库同步、PR 管理、代码推送 |

### 核心机制：Agent ID 调用

主 Agent 不直接执行子任务，而是通过 Agent ID 把任务"派发"给对应的专职 Agent：

```
用户 → 课程开发 Agent（主调度）
                    ↓ 通过 ID 调用
            ┌───────┼───────┐
            ↓       ↓       ↓
        版本管理  开发   运维 Agent
        Agent    Agent
```

具体操作方式：在课程开发 Agent 的对话中，直接告诉它其他 Agent 的 ID：

```
019f8f00-488f-7990-8f3e-cc95cb8b1395 这是版本管理 Agent
019f6724-20e1-7910-a101-9e57a2a8b105 这是开发 Agent
后面对应你安排给他们，然后去找他们做就行
```

主 Agent 收到 ID 后，理解任务需求，自行决定调用哪个 Agent、传递什么上下文，执行完毕后汇总结果返回。

## 实际效果

老师在课堂上演示了一个完整流程：

1. **用户向课程开发 Agent 发出指令**：更新 day07 的学生版和老师版课件
2. **课程开发 Agent 分析任务**：需要改文件 → 调用开发 Agent；需要提交版本 → 调用版本管理 Agent
3. **各 Agent 执行子任务**：开发 Agent 修改了 `day07-student.html`、`day07-teacher.html`、`Day07-完整课程设计.md` 等 7 个文件；版本管理 Agent 完成提交，生成变更记录
4. **结果汇总**：课程开发 Agent 返回最终结果，包含学生入口、老师版 PPT、课件包等链接，以及 git 风格的文件变更记录（+6 -5 等）

整个过程用户只发了一条指令，后续分工和协调全部由主 Agent 完成。

## 为什么这个模式好

### 1. 上下文不丢

主 Agent 知道整个项目的来龙去脉。当它去调版本管理 Agent 的时候，能把任务的背景信息一并带过去——不用人工重新解释"这个项目是什么、为什么要改这些文件"。

### 2. 管理成本低

用户只需要跟一个 Agent 说话。不用自己记住"版本管理该找谁、部署该找谁"，主 Agent 自己知道该调谁。

### 3. 项目内闭环

所有 Agent 都在同一个项目下面，ID 互相可见。不会出现跨项目找不到 Agent 的情况。项目维度天然就是一个协作边界。

### 4. 职责清晰

每个 Agent 只做自己擅长的事。课程开发的不用关心 git 操作，版本管理的不用关心课件内容。单一职责，出错容易定位。

## 与"单 Agent 全干"的对比

| 维度 | 单 Agent 全干 | 主 Agent 调度多 Agent |
| --- | --- | --- |
| 上下文管理 | 全压在一个 Agent 上，容易超长 | 各 Agent 各管各的上下文，互不干扰 |
| 专业能力 | 一个 Agent 啥都做，深度不够 | 每个 Agent 专注自己的领域 |
| 出错定位 | 不知道哪一步出了问题 | 按 Agent 粒度定位，清楚是谁的锅 |
| 用户负担 | 记住所有操作步骤 | 只管发指令，主 Agent 自己分工 |
| 扩展性 | 加新功能只能往一个 Agent 里塞 | 新增一个专职 Agent，给它一个 ID |

## 可迁移场景

这个模式不局限于课程开发，任何需要多步骤协作的项目都适用：

- **游戏开发**：主调度 Agent 持有前端开发 Agent、游戏逻辑 Agent、部署运维 Agent 的 ID
- **内容生产**：主调度 Agent 持有写作 Agent、配图 Agent、排版 Agent、发布 Agent 的 ID
- **数据分析**：主调度 Agent 持有数据采集 Agent、清洗 Agent、可视化 Agent 的 ID

关键就一句话：**选一个主 Agent，把其他 Agent 的 ID 给它，让它自己分工。**

## 前提条件

- 平台需要支持"通过 ID 调用其他 Agent"的能力（Kimi Work 支持）
- Agent 需要在同一个项目下，ID 互相可见
- 主 Agent 需要有足够的理解力来拆分任务和派发指令

## 经验总结

之前一直觉得 Agent 就是"一个聪明的助手"，给它一个任务它自己干。今天的模式让我意识到——Agent 可以是一个**调度者**，它不一定要自己动手，可以通过 ID 调用其他 Agent 来分工协作。

这个思维的转变比具体操作更重要：

> **从"我跟每个 Agent 分别对话"变成"我告诉主 Agent 要做什么，它自己去协调"。**

本质上是从"单兵作战"升级到"团队编排"。以前是我自己在多个 Agent 之间当传话筒，现在让主 Agent 替我当这个传话筒——而我只需要说一句"做什么"。
