# 从每次输密码到一键部署：我的自动化部署启蒙记

> 作者：kim ｜ 安吉集训过程记录

## 手动时代：一套动作，几十遍

做第一个项目的时候，我的部署流程是纯手动的：改完代码，手动把文件传到服务器，ssh 登录上去（输一次密码），重启服务，打开浏览器验证。发现 bug，再从头来一遍。

这套动作我操作了几十遍。每一遍都要输入密码，每一遍都要祈祷没有手滑。当时我觉得这就是开发的常态——直到"1 分钟改 5 分钟"这件事，把这种常态的代价彻底暴露了出来。

## 事故：一个改不掉的参数

需求很简单：把游戏的单把时长从 60 秒改成 300 秒。就一行常量的事。

但我改完、上传、重启，进游戏一看：还是 1 分钟。

我以为没传上去，又传一遍。还是 1 分钟。

我以为缓存问题，强刷浏览器。还是 1 分钟。

我前前后后部署了好几遍，每次都是熟悉的配方：输密码、上传、登录、输密码、重启。每一遍都"成功"了——文件传上去了，进程也在跑——但游戏就是旧版本。

## 真相：代码从来没问题上，是旧进程根本没死

排查后发现，问题出在一个谁也想不到的地方。

重启服务的命令是 `pkill -f "node server.js"`，意思是"杀掉叫 node server.js 的进程"。但服务器上那个旧进程，启动命令是这样的：

```bash
node --trace-uncaught --unhandled-rejections=strict server.js
```

中间夹着两个参数。`pkill` 的模式是"node server.js"，**对不上**，所以一次都没杀掉它。

于是形成了一个完美的骗局：

1. 代码每次都成功上传到了服务器 ✓
2. 杀进程的命令每次都执行了，但什么也没杀到 ✗
3. 新进程启动失败（端口被旧进程占着，EADDRINUSE），悄悄退出 ✗
4. 旧进程继续愉快地运行着旧代码，接受所有访问 ✓

我看到的每一分钟，都是那个几小时前启动的旧进程发给我的。

## 解法：把流程固化成脚本，把密码变成钥匙

痛定思痛，这次做了三件事。

**第一件：SSH 密钥免密。** 在本机生成一对密钥，公钥装到服务器上。从此以后，上传、登录、执行命令全部免密。这是自动化的前提——密码交互没了，脚本才能从头跑到尾不卡壳。

**第二件：部署脚本。** 把整套流程固化成一个脚本 `deploy-tank.sh`，以后部署只需要敲一条命令：

```bash
bash ~/deploy-tank.sh
```

脚本自动完成四步：

1. `rsync` 增量同步代码（只传改动过的文件）
2. ssh 到服务器上校验磁盘文件确实是新版本（grep 特征代码行）
3. 精确杀死旧进程（`pkill -f 'server.js'`，这次不带 `node` 前缀，带参数也能匹配），确认端口清空后重启
4. 从外部验证：真的建一个游戏房间，读 `roundTimeLeft`，确认返回的是 300

**第三件：行为验证写进流程。** 这是最重要的一步。以前所有失败都源于"以为部署成功了"——文件传了、进程活着，但跑的是旧代码。所以脚本最后一步不是检查"活着没"，而是检查"行为对不对"。返回 300 才算完，返回 60 当场报警。

## 现在的日常

改代码 → `bash ~/deploy-tank.sh` → 等十几秒 → 看到 `roundTimeLeft = 300` → 收工。

没有密码，没有断行，没有僵尸进程，没有"不知道处于什么状态"。甚至 agent 也可以直接替我执行这个脚本——密钥配好后，它能自己 ssh 上去排查问题、自动修复。

## 复盘：那几十遍手动操作，浪费在哪

回头看，手动时代一半的工夫不是花在"操作"上，而是花在**不知道自己处于什么状态**上：

- 不知道代码传没传上去（传是传了，但传对目录了吗？）
- 不知道旧进程杀没杀掉（命令执行了，但匹配到了吗？）
- 不知道现在跑的是新是旧（进程活着，但活的是谁？）

手动时代的每一步都没有成功判据，失败不会当场暴露，而是攒到打开游戏那一刻才爆发。自动化脚本的真正价值，不是"少敲几条命令"，而是**每一步都有明确的成功标准，哪一步出问题当场就知道**。

至于技术本身——rsync、ssh 密钥、nohup，都是用了十几年的标准工具，没有任何新发明。但对我来说，"每次手动输密码的时代"确确实实结束了。这大概就是工程能力的真相：不是会多少新技术，而是把踩过的坑，变成再也踩不到的流程。
